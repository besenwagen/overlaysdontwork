What this is

First, an apology for any users with disabilities. This page is purposely broken in ways that will frustrate you in weird and unpredictable ways. Here's why:

For years, I've been warning people not to get sucked into the false promises of products that claim to magically make your website accessible:

  • "On overlays as a means of resolving website accessibility issues"
  • "Can assistive technology make a website accessble?"
  • "When the treatment is worse than the disease."

I'm not the only one with these criticisms. My feelings have been echoed by many others including Jim Thatcher a legend in accessibility.

Unfortunately, well-meaning customers are still being suckered by sales people claiming that their product "automatically transforms the website to meet 508 and WCAG 2.0 AA accessibility standards." These claims are, in a word: bullshit. I've developed this page to prove this.

This page is a challenge to all of those vendors: Make this page accessible by CSUN 2018 and prove it.

How it Works

This page presents a large number of examples of inaccessible code that cannot be reliably fixed by an "overlay" product. Each of the examples is provided along with an explanation of why it is a problem and why an overlay cannot fix it.

The problems are chosen at random and presented via JavaScript. This, of course, is problem #1 which is that all of the content inside the <main> element relies on JavaScript. Then again, so do the overlays.

Some overlay products need to be explicitly coded to the specific web site, which makes the overlay more effective, but also more brittle. Custom overlay solutions are quick to break when sites are changed during normal maintenance and modifications. I've taken steps to confuse attempts at custom solutions, too.

Although some of these examples might seem like weird edge-cases, they are not. Every one is an example based on the most commonly logged issues across over 260 accessibility audits that included manual testing.
